# 模型厂商调用适配层

## 动机

agent 系统底层需要调用不同厂商的模型（OpenAI / Anthropic / Google / Bedrock / Mistral ...），
每家 API 的协议、鉴权、流式格式都不同。
若直接耦合，上层业务逻辑会被厂商差异污染，切换模型代价极高。

## 参考实现：pi-mono/packages/ai

### 核心抽象

**`Model<TApi>`**（`types.ts`）
- 统一的模型描述对象：id、name、api、provider、baseUrl、cost、contextWindow 等
- `api` 字段是关键判别符，标识使用哪套 API 协议（如 `"anthropic-messages"` / `"openai-completions"`）

**`StreamFunction<TApi>`**（`types.ts`）
```ts
type StreamFunction<TApi, TOptions> = (
  model: Model<TApi>,
  context: Context,
  options?: TOptions,
) => AssistantMessageEventStream;
```
- 所有 provider 实现同一个函数签名
- 输入：模型 + 上下文（system prompt + messages + tools）+ 选项
- 输出：统一的 `AssistantMessageEventStream`（事件流）

**`ApiProvider<TApi>`**（`api-registry.ts`）
- 每个厂商注册一个 `{ api, stream, streamSimple }` 对象

**`ApiProviderRegistry`**（`api-registry.ts`）
- 全局 `Map<api-string, ApiProvider>`
- 支持 `registerApiProvider` / `getApiProvider` / `unregisterApiProvider`
- `sourceId` 机制支持按来源批量注销（可插拔）

### 已支持的 Provider

| Provider | API 协议 |
|---|---|
| Anthropic | `anthropic-messages` |
| OpenAI | `openai-completions` / `openai-responses` |
| Google | `google-generative-ai` / `google-vertex` / `google-gemini-cli` |
| Amazon Bedrock | `bedrock-converse-stream` |
| Mistral | `mistral-conversations` |
| Azure OpenAI | `azure-openai-responses` |
| GitHub Copilot | 复用 openai-responses 协议 |
| 及 xai/groq/openrouter 等 OpenAI 兼容厂商 | `openai-completions` compat 模式 |

### 消息格式统一

`Context` 统一了上层消息格式：
```ts
interface Context {
  systemPrompt?: string;
  messages: Message[];  // UserMessage | AssistantMessage | ToolResultMessage
  tools?: Tool[];
}
```
各 provider 内部负责将 `Context` 转成自己的 API payload（`transform-messages.ts` 等）。

### 流式事件协议

所有 provider 输出统一的事件序列：
`start → text_delta* → toolcall_* → done/error`

## 设计要点

1. **面向接口**：上层只调用 `getApiProvider(model.api).stream(model, context, options)` 
2. **可插拔**：通过 `registerApiProvider` 运行时注册，`unregisterApiProviders(sourceId)` 热卸载
3. **类型安全**：泛型 `TApi` 确保 model 与 provider 的 api 字段一致，编译期防止错配
4. **OpenAI 兼容层**：`OpenAICompletionsCompat` 收纳各厂商的细节差异，避免污染核心接口

## 深挖：鉴权层设计

### 现状（pi-mono 的做法）

鉴权逻辑分两层：

**1. 环境变量解析（`env-api-keys.ts`）**

`getEnvApiKey(provider)` 统一从环境变量读取 key，屏蔽了各厂商的变量名差异：

```
anthropic  → ANTHROPIC_OAUTH_TOKEN 优先，fallback ANTHROPIC_API_KEY
openai     → OPENAI_API_KEY
google-vertex → GOOGLE_CLOUD_API_KEY 或 ADC 文件检测 → 返回 "<authenticated>"
amazon-bedrock → AWS_PROFILE / IAM keys / Bearer Token / ECS role … → 返回 "<authenticated>"
github-copilot → COPILOT_GITHUB_TOKEN / GH_TOKEN / GITHUB_TOKEN
```

注意：Vertex / Bedrock 这类"非 key 鉴权"的 provider，返回哨兵字符串 `"<authenticated>"` 表示"已就绪但没有真实 key"，让上层逻辑统一判断。

**2. 客户端构建（以 Anthropic 为例，`anthropic.ts → createClient()`）**

provider 内部根据 key 类型分三种路径：
```
options.client 已提供    → 直接用（支持 AnthropicVertex 等注入）
isOAuthToken(key)        → Bearer 鉴权 + Claude Code identity headers（stealth 模式）
普通 API key             → SDK 标准 apiKey 字段
```

OAuth token 识别：`key.includes("sk-ant-oat")` — 字符串前缀判断。

GitHub Copilot 路径：同样 Bearer，但 headers 不同（无 claude-code identity，有 Copilot 专属 beta headers）。

### 设计问题与改进方向

**现状耦合点**：鉴权逻辑散在每个 provider 文件内，`getEnvApiKey` 只是减少了"读哪个环境变量"的重复，但"如何用这个 key 构建客户端"完全由 provider 自己决定。

**如果要进一步解耦**，可以引入独立的 `AuthProvider` 接口：
```ts
interface AuthProvider {
  getCredential(provider: Provider): Promise<Credential>
}
type Credential =
  | { type: "apiKey"; key: string }
  | { type: "bearer"; token: string }
  | { type: "adc" }           // Google Application Default Credentials
  | { type: "aws"; ... }      // AWS SDK managed
```
让鉴权成为可注入的依赖，而不是 provider 内部硬编码，便于：
- 单元测试时 mock 凭证
- 运行时动态轮换 key（多租户场景）
- 统一的 OAuth 刷新逻辑

## 深挖：缓存命中机制

### Anthropic Prompt Cache 原理

Anthropic 的 prompt cache 是**服务端 KV 缓存**，按前缀匹配：
- 相同的 system prompt + messages 前缀 → cache hit，只计 `cache_read_input_tokens`（低价）
- 新内容写入缓存 → `cache_creation_input_tokens`（稍高于普通输入价）

### pi-mono 的实现（`anthropic.ts`）

**触发方式**：在 API payload 的特定位置注入 `cache_control: { type: "ephemeral" }` 字段。

**三个注入点**：
```
1. system prompt 块           → 缓存系统提示（变化最少，命中率最高）
2. tools 列表最后一个工具     → 缓存工具定义（agent 场景工具列表固定）
3. messages 最后一条 user msg → 缓存历史对话（多轮对话节省重复 token）
```

代码关键逻辑（`buildParams` + `convertMessages` + `convertTools`）：
```ts
// 1. system prompt
params.system = [{ type: "text", text: systemPrompt, cache_control: cacheControl }]

// 2. tools（最后一个工具加标记）
tools.map((tool, index) => ({
  ...tool,
  ...(cacheControl && index === tools.length - 1 ? { cache_control: cacheControl } : {})
}))

// 3. 最后一条 user message 的最后一个 content block
lastBlock.cache_control = cacheControl
```

**TTL 控制**（`getCacheControl()`）：
```
retention === "none"  → 不注入 cache_control（不缓存）
retention === "short" → { type: "ephemeral" }（默认 5 分钟 TTL）
retention === "long"  → { type: "ephemeral", ttl: "1h" }（仅 api.anthropic.com 支持）
```

**用量上报**（`message_start` / `message_delta` 事件）：
```ts
output.usage.cacheRead  = event.message.usage.cache_read_input_tokens
output.usage.cacheWrite = event.message.usage.cache_creation_input_tokens
```
最终计入 `AssistantMessage.usage`，供上层监控缓存效果。

### 设计要点总结

| 问题 | pi-mono 的答案 |
|---|---|
| 何时缓存？ | 调用方传 `cacheRetention` 选项（none/short/long） |
| 缓存哪里？ | system + 最后一个 tool + 最后一条 user msg（三点锚定） |
| 如何感知命中？ | `usage.cacheRead > 0` |
| 多轮对话优化 | 每轮都把最新 user msg 打上 cache_control，下轮命中历史 |

### 对自己 agent 项目的启示

- 缓存逻辑应该在 **provider 适配层内部**处理，上层只需传语义化的 `cacheRetention`
- 建议把 `cacheRead` / `cacheWrite` token 统计暴露给上层，让 agent 框架可以做成本监控
- 如果 agent 有固定工具集，务必缓存 tools（多轮对话收益最大）

## 三厂商缓存机制横向对比

### Anthropic：主动注入式（显式标记）

**机制**：调用方在 payload 的特定 block 上附加 `cache_control: { type: "ephemeral" }` 字段，
由服务端在该位置建立缓存断点（前缀匹配）。

**注入点**（`buildParams` + `convertMessages` + `convertTools`）：
```
system prompt block     → cache_control
tools 列表最后一个工具  → cache_control
messages 最后一条 user msg 的最后一个 block → cache_control
```

**用量字段**（`message_start` / `message_delta` 事件）：
```
cache_read_input_tokens    → usage.cacheRead
cache_creation_input_tokens → usage.cacheWrite
```

**TTL**：short = 5分钟（默认），long = 1小时（仅 api.anthropic.com）

---

### OpenAI：自动式（服务端透明缓存）

**机制**：OpenAI 服务端**自动**对超过 1024 tokens 的 prompt 前缀做缓存，
调用方**无需**任何标记。缓存粒度以 128 token 为单位对齐。

**pi-mono 的处理（`parseChunkUsage`）**：
```ts
// 从流式 chunk 的 usage 字段读取命中数据
prompt_tokens_details.cached_tokens       → 原始命中数
prompt_tokens_details.cache_write_tokens  → 写入数（部分厂商上报）

// 归一化：某些 OpenRouter 兼容厂商 cached_tokens 包含 write，需去重
cacheRead = cacheWriteTokens > 0
  ? Math.max(0, reportedCachedTokens - cacheWriteTokens)
  : reportedCachedTokens

// 净输入 token（真正计费的新 token）
input = promptTokens - cacheRead - cacheWrite
```

**特殊情况**：OpenRouter 代理 Anthropic 模型时，会退化为 Anthropic 的显式注入方式
（`maybeAddOpenRouterAnthropicCacheControl`），自动在最后一条 user/assistant message
的最后一个 text part 上追加 `cache_control: { type: "ephemeral" }`。

**TTL**：OpenAI 官方约 5~10 分钟（服务端决定，文档未明确）

---

### Google Gemini：上下文缓存（Context Caching API，与前两者完全不同）

**机制**：Google 的 prompt cache 是**独立的 Context Caching API**——需要提前创建一个
`cachedContent` 资源，获取 `cachedContentName`，再在请求中引用它。
这套机制在 pi-mono 的 provider 层**完全没有实现**，设计上直接绕过了。

**pi-mono 实际做的事**（`google.ts` 的 `usageMetadata` 处理）：
```ts
// 被动上报：如果上游返回了 cachedContentTokenCount，直接记录
cacheRead = chunk.usageMetadata.cachedContentTokenCount || 0
input     = promptTokenCount - cachedContentTokenCount  // 净输入
cacheWrite = 0  // 始终为 0，因为没有主动写缓存
```

即：Google provider 只做了"**读取缓存命中上报**"，没有主动创建缓存的逻辑。
这意味着如果调用方自己在 Google API 侧预建了 `cachedContent`，命中数据仍能被正确统计，
但 pi-mono 本身不触发缓存写入。

## 三者对比总结

| 维度 | Anthropic | OpenAI | Google |
|---|---|---|---|
| 缓存触发方式 | **主动**：payload 里注入 `cache_control` | **自动**：服务端透明处理 | **预建**：Context Caching API，单独创建资源 |
| 调用方需要操作 | 在正确位置打标记 | 无需操作 | 提前调用 Caching API，请求中引用 cachedContentName |
| pi-mono 实现程度 | ✅ 完整实现（注入 + 上报） | ✅ 上报（自动缓存无需注入） | ⚠️ 仅上报，未实现主动缓存写入 |
| 缓存粒度 | 任意断点（按注入位置） | 1024 token 前缀，128 token 对齐 | 按 cachedContent 资源定义 |
| TTL | 5分钟（short）/ 1小时（long） | ~5-10分钟（服务端决定） | 可配置（默认 1小时，最长 30天） |
| 感知命中 | `usage.cacheRead > 0` | `usage.cacheRead > 0` | `usage.cacheRead > 0` |
| 跨轮多次命中 | ✅（每轮打新断点，前缀命中） | ✅（自动） | ✅（资源复用） |

## 适配层统一的设计结论

**好消息**：上层 `AssistantMessage.usage` 已经统一了三者的输出——
无论底层机制多么不同，上层都只看 `cacheRead` / `cacheWrite`，接口一致。

**坏消息**：**写入侧**没有统一。调用方如果想主动控制缓存：
- Anthropic 需要传 `cacheRetention` 选项
- OpenAI 不需要任何操作（自动）
- Google 需要在 provider 外部自己管理 `cachedContent` 资源的生命周期

**如果要在自己的 agent 项目里统一缓存控制**，推荐分两层：

```
上层语义（agent 感知）：
  CachePolicy = "none" | "auto" | "aggressive"

适配层转换（per-provider）：
  Anthropic → 注入 cache_control（短/长 TTL）
  OpenAI    → 无操作（already auto）
  Google    → 若需主动缓存，调用 Context Caching API 并注入 cachedContentName
```

这样上层 agent 只需声明"我要激进缓存"，底层各自落地。
