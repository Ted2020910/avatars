# agent 循环与 tool use 层解耦

## 参考实现：pi-mono/packages/agent

### 整体分层

```
┌─────────────────────────────────────┐
│  Agent（有状态外壳）                 │  ← state + 生命周期 + 事件订阅
│  agent.ts                           │
├─────────────────────────────────────┤
│  runLoop / agentLoop（无状态循环）   │  ← 纯函数，emit 事件流
│  agent-loop.ts                      │
├─────────────────────────────────────┤
│  AgentTool（工具定义）               │  ← 与 LLM Tool Schema 解耦
│  types.ts                           │
├─────────────────────────────────────┤
│  StreamFn（LLM 调用接口）            │  ← = 002 层的 streamSimple
│  types.ts                           │
└─────────────────────────────────────┘
```

---

### 第一解耦：AgentMessage vs LLM Message（消息类型隔离）

**问题**：agent 的对话历史里不只有 LLM 消息——还有 UI 通知、状态更新、自定义消息类型。
但 LLM API 只接受 `UserMessage | AssistantMessage | ToolResultMessage`。

**做法**：引入 `AgentMessage` 作为 agent 内部流通的宽类型：
```ts
// 可扩展：通过 declaration merging 注入自定义类型
interface CustomAgentMessages {
  artifact: ArtifactMessage;       // 应用扩展
  notification: NotificationMessage;
}
type AgentMessage = Message | CustomAgentMessages[keyof CustomAgentMessages];
```

**边界转换**：在每次 LLM 调用前，通过 `convertToLlm` 函数做一次转换：
```ts
// AgentMessage[] → Message[]（只在 LLM 调用边界发生一次）
const llmMessages = await config.convertToLlm(messages);
```
这个函数是**可注入**的，业务方决定哪些 AgentMessage 过滤掉、哪些转成 user 消息。

**另一个钩子**：`transformContext` 在 `convertToLlm` 之前运行，在 AgentMessage 层操作：
```ts
// 适合做：context window 裁剪、注入外部上下文
transformContext: async (messages) => pruneIfOverLimit(messages)
```

两个钩子分工明确：`transformContext` 在 agent 语义层剪裁，`convertToLlm` 做格式转换。

### 第二解耦：AgentTool vs LLM Tool Schema（工具定义隔离）

**LLM 层的 `Tool`**（来自 002 层）只有 schema 描述：
```ts
interface Tool<TParameters extends TSchema> {
  name: string;
  description: string;
  parameters: TParameters;
}
```

**agent 层的 `AgentTool`** 在此之上扩展了运行时语义：
```ts
interface AgentTool<TParameters, TDetails> extends Tool<TParameters> {
  label: string;              // UI 展示用
  prepareArguments?: (args) => Static<TParameters>;  // 参数预处理/兼容 shim
  execute: (                  // 实际执行逻辑
    toolCallId: string,
    params: Static<TParameters>,
    signal?: AbortSignal,
    onUpdate?: AgentToolUpdateCallback,  // 流式进度回调
  ) => Promise<AgentToolResult<TDetails>>;
  executionMode?: "sequential" | "parallel";  // 并发控制
}
```

**调用流程**（`agent-loop.ts`）：
```
LLM 返回 toolCall
    ↓
prepareToolCall()
  ├─ 查找工具（找不到 → 立即错误结果）
  ├─ prepareArguments()（参数 shim）
  ├─ validateToolArguments()（schema 验证）
  └─ beforeToolCall()（可拦截：return { block: true }）
    ↓
executePreparedToolCall()
  └─ tool.execute()（真正执行，支持 onUpdate 流式进度）
    ↓
finalizeExecutedToolCall()
  └─ afterToolCall()（可改写结果：content / details / isError）
    ↓
ToolResultMessage → 追加到 context.messages → 下一轮 LLM
```

**两个拦截钩子**：
- `beforeToolCall`：执行前，可阻断（权限控制、危险操作确认）
- `afterToolCall`：执行后，可改写结果（结果截断、格式转换、日志注入）

**工具执行模式**（`toolExecution`）：
- `"parallel"`（默认）：所有工具 prepare 完成后并发 execute，结果按原始顺序 emit
- `"sequential"`：逐个执行
- 工具可通过 `executionMode: "sequential"` 声明自己必须串行（如文件写操作）

### 第三解耦：事件系统（loop 与 UI/状态 解耦）

`runLoop` 是一个**纯函数**，不持有任何状态，通过 `emit(AgentEvent)` 向外广播：

```ts
// 生命周期事件完整列表
type AgentEvent =
  | { type: "agent_start" }
  | { type: "agent_end"; messages: AgentMessage[] }
  | { type: "turn_start" }
  | { type: "turn_end"; message; toolResults }
  | { type: "message_start"; message }
  | { type: "message_update"; message; assistantMessageEvent }  // 流式 delta
  | { type: "message_end"; message }
  | { type: "tool_execution_start"; toolCallId; toolName; args }
  | { type: "tool_execution_update"; ... partialResult }       // 工具流式进度
  | { type: "tool_execution_end"; toolCallId; result; isError }
```

**`Agent` 类**消费这些事件做两件事：
1. **reducer**：更新 `_state`（isStreaming / streamingMessage / pendingToolCalls / messages）
2. **广播**：依次 await 所有 `subscribe()` 监听器

UI 层、日志层、测试层都只订阅事件，**零耦合于循环内部**。

### 第四解耦：Steering / Follow-up 消息队列

**问题**：agent 运行中，用户可能要"中途插话"或"追加任务"，但不能直接往 context 里塞消息。

**做法**：两个独立队列，通过回调注入循环：
```ts
getSteeringMessages: async () => steeringQueue.drain()
// 在每轮 tool calls 执行完、下次 LLM call 之前消费
// 用途：运行中转向（"先停下来，改做这个"）

getFollowUpMessages: async () => followUpQueue.drain()
// 在 agent 即将退出时消费（没有 tool calls 也没有 steering）
// 用途：追加后续任务（"做完以后再做这个"）
```

循环结构（双层 while）：
```
外层 while(true)          ← follow-up 驱动
  内层 while(hasToolCalls || pendingMessages)  ← steering + tool 驱动
    inject steering msgs
    LLM call
    execute tool calls
    poll steering
  poll follow-up
  if no follow-up → break
```

### 跨轮消息上下文处理（transform-messages.ts）

agent loop 在 LLM call 边界调用 `convertToLlm`，后者最终走到 `transformMessages`：

1. **thinking block 清理**：跨模型/provider 时，thinking block 降级为普通文本（防止 API 拒绝）
2. **tool call ID 归一化**：不同 provider 的 ID 格式各异（OpenAI Responses 生成 450+ chars），在此截断/转义
3. **孤儿 tool call 填充**：assistant 发出 tool call 但 context 里没有对应 toolResult 时，自动插入 `{ isError: true, content: "No result provided" }` 占位，防止 API 报错
4. **错误/中断 assistant 消息剪掉**：`stopReason === "error" | "aborted"` 的 assistant 消息不回放给 LLM

## 设计要点总结

| 解耦维度 | 手段 | 效果 |
|---|---|---|
| agent 消息 vs LLM 消息 | `convertToLlm` + `transformContext` 钩子 | 自定义消息类型透明流通 |
| 工具定义 vs 工具执行 | `AgentTool` 扩展 `Tool`，execute 独立 | 工具可替换，schema 不变 |
| 循环逻辑 vs 状态/UI | 纯函数 `runLoop` + `emit` 事件流 | 订阅者零耦合 |
| 同步控制流 vs 异步注入 | steering / follow-up 队列 | 运行中可安全注入消息 |
| 跨模型兼容性 | `transformMessages` 在边界归一化 | 历史消息跨 provider 安全回放 |

## 对自己 agent 项目的启示

- **`AgentMessage` 宽类型**是关键设计——UI 消息、日志消息、状态消息都能放进 transcript，
  `convertToLlm` 负责过滤，LLM 只看它该看的
- **`beforeToolCall` / `afterToolCall`** 是最值得保留的两个钩子：权限控制和结果后处理是 agent 框架的核心需求
- **事件驱动**比回调更易组合：subscribe 可以叠加多个监听器（日志 + UI + 测试），互不干扰
- **steering queue** 解决了"运行中转向"这个棘手问题，不需要 abort + 重建
