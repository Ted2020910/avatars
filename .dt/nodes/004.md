# convertToLlm 与 AgentMessage 自定义类型设计

## 真实代码中的调用位置

`agent-loop.ts` 第 289 行，在 `streamAssistantResponse()` 里：
```ts
// 先 transformContext（AgentMessage 层剪裁）
let messages = context.messages;
if (config.transformContext) {
    messages = await config.transformContext(messages, signal);
}
// 再 convertToLlm（格式转换，过滤掉 LLM 不理解的消息）
const llmMessages = await config.convertToLlm(messages);
```

`agent.ts` 默认实现（第 28-32 行）只保留三种标准消息：
```ts
function defaultConvertToLlm(messages: AgentMessage[]): Message[] {
    return messages.filter(
        (message) => message.role === "user" || message.role === "assistant" || message.role === "toolResult",
    );
}
```

## 需要决策的问题

### 1. 我们的系统有哪些自定义消息类型？

通过 declaration merging 扩展 `CustomAgentMessages`：
```ts
declare module "@earendil-works/pi-agent" {
    interface CustomAgentMessages {
        // 在这里注入我们自己的类型
    }
}
```

候选自定义消息类型（待确认）：
- `notification`：UI 状态通知（不发给 LLM，只给前端渲染）
- `artifact`：agent 产出的结构化结果（文件、代码片段）——要不要回放给 LLM？
- `system_update`：运行时系统提示变更记录——不发给 LLM，但需要持久化到 session

### 2. transformContext 里做什么？

两种策略选择：
- **token 估算裁剪**：如果 `estimateTokens(messages) > threshold`，从最早的 user/assistant 对开始删除，保留最近 N 轮
- **delegation to compaction**：不在 transformContext 裁剪，依赖 AgentHarness.compact() 做离线压缩

pi-mono 的 AgentHarness 选择了后者：transformContext 里只做 hook 转发（`agent-harness.ts` 第 204 行），真正的 compaction 是外部调用 `harness.compact()`。
